## Introduction
Probabilistic Diffusion Models Framework – A modular and extensible port of Diffusion Probabilistic Models (DDPM), originally proposed by () along with OpenAI’s improved version.

Built on Lightning, enables rapid experimentation, seamless integration with Hydra, and advanced logging offering a flexible and scalable environment for diffusion model development and experimentation.

📖 Comprehensive Guide: Accompanying this framework is a detailed script that meticulously explains every equation, derivation, and implementation detail, ensuring a deep understanding of the models, their underlying principles, and potential challenges.

🧪 Reproducible Experiments: The repository also provides a curated set of experiments, complete with checkpoints and instructions on sampling, resuming training, and fine-tuning, allowing users to explore and extend the models with ease. 🚀

🧩 Unit Tests for Core Components: To ensure correctness and provide deeper insights into the framework’s key elements, we include comprehensive unit tests covering critical components such as loss functions, model parameterizations, and sampling steps. These tests not only verify implementation integrity but also serve as a valuable tool for understanding individual building blocks, making it easier to experiment, debug, and extend with higher confidence. ✅

## Installation
This repository uses **uv** as the package manager for efficient and reproducible dependency management.

### Install `uv`
First, install `uv` with standalone installer:

```sh
curl -LsSf https://astral.sh/uv/install.sh | sh
```

Or from PyPi
```sh
pip install uv
```

### Clone repository
Next, clone the repository and navigate into it:
```sh
git clone https://github.com/PiotrGrzybowski/diffusion.git
cd diffusion
```

### Create an environment and install dependencies
Create a virtual environment and install all required dependencies:

```sh
uv venv
source .venv/bin/activate
uv install
```

### Run tests to verify installation
To ensure everything is set up correctly, run the test suite:

```sh
pytest tests
```

You're all set! 🚀 Happy experimenting!


## Basic Usage
This framework provides two main functionalities:
- `train.py` – Runs the training process.
- `sample.py` – Generates samples from a trained model.

Both scripts **share the same configuration system**, allowing seamless control over diffusion settings, model architecture, and logging.  

### **Training a Model**
To start training with the **default configuration**, run:
```bash
uv run train.py
```
By default, this runs training on `MNIST` for 10 epochs using $L_{\text{simple}}$ loss with `epsilon` mean parametrization, as this setup provides quick and stable convergence.


#### Outputs
Hydra creates new output directory for every executed run. Default logging structure is as follows. It contains logs, checkpoints and images sampled during the training at specyfic epochs.

```
├── logs
    ├── task_name
        ├── runs                            # Logs generated by single runs
            ├── YYYY-MM-DD_HH-MM-SS         # Datetime of the run
            │   ├── .hydra                  # Hydra logs
            │   ├── csv                     # Csv logs
            │   ├── wandb                   # Weights&Biases logs
            │   ├── checkpoints             # Training checkpoints
            │   ├── images                  # Sampled images at checkpoints
            │   └── ...                     # Any other thing saved during training
            └── ...
```


### Sampling from Model
To generate samples use: 

```bash
uv run sample.py ckpt_path=<path_to_checkpoint>
```
You must specify a trained checkpoint (ckpt_path) to ensure meaningful outputs. If no checkpoint is provided, sampling will proceed with random weights, leading to meaningless results.


## Configuration

This repository is designed with a structured and modular Hydra-based configuration system. Below is an overview of the key configuration directories and their purpose.

### MLOps Configuration
The `MLOps` configuration manages dataset handling, logging, training parameters, and experiment tracking, ensuring reproducibility and seamless configuration for different workflows.

| Category               | Config Path       | Description | Options |
|-----------------------|------------------|-------------|---------|
| Data | `data/`          | Defines data modules for loading datasets. | Supports MNIST and CIFAR-10, with options to extend. |
| Logging          | `logger/`        | Configures logging and experiment tracking. | Supports CSV, Weights & Biases (wandb), and TensorBoard. |
| Trainer | `trainer/`    | Controls the core training parameters. | Number of epochs, devices (CPU/GPU), precision settings, and gradient accumulation. |
| Main Training Config  | `train.yaml`  | The root configuration that integrates all components and settings. | Entry point for model training. |

Other directories like `extras/`, `paths/`, and `hydra/` exist but are not essential for everyday use, primarily handling path resolution and Hydra job management.


### **Default Training Configuration** 
Please find the important defaults in `train.yaml` file
```yaml
_defaults_:
  - data: mnist
  - trainer: cpu
    ...

in_channels: 1   # Number of input channels (for RGB images, set to 3).
out_channels: 1  # Determines whether variance is trainable. Set to 2*in_channels to enable it.
dim: 28          # Image size (e.g., 28 for MNIST, 32 for CIFAR-10, 64 for ImageNet-64).
timesteps: 1000  # Number of diffusion steps.
...
```
Key global parameters—such as image size, input channels, and diffusion steps —can be directly overridden in the CLI. As an example consider experiment for `CIFAR10` dataset, to run on single `gpu`. To override these values when running an experiment:

```bash
uv run train.py data=mnist in_channels=3 dim=32 trainer=gpu
```

### Diffusion Module Configuration
The diffusion configuration is designed for maximum flexibility, allowing control over multiple components. This structure supports rapid experimentation, making it easy to tweak individual components while maintaining reproducibility. Those configs are stored under `configs/diffusion`. See the overview in the table below.

| Category        | Config Path                  | Description | Options |
|-----------------|--------------------------|-----------------|---------|
| **Noise Scheduler** | `scheduler/` | Defines strategies how noise schedule is defined. | `linear`, `cosine`|
| **Mean Strategy** | `mean/` | Defines strategies for modeling the mean. | `direct`, `epsilon`, `xstart` |
| **Variance Handling** | `variance/` | Different strategies for modeling the variance. | `direct`, `fixed_large`, `fixed_small`, `trainable_range`  |
| **Loss Functions** | `loss/` | Different objectives for training the model. | `mse_mean_direct/simple`, `mse_mean_epsilon/simple`, `mse_mean_xstart/simple`, `vlb`, `hybrid`  |
| **Model Architecture** | `model/` |  Different U-Net configurations for denoising.  | `small_unet`, `unet` |

#### **Default Diffusion Configuration**
```yaml
defaults:
  - _self_
  - scheduler: linear                           # Noise Scheduling
  - mean@posterior_mean: xstart                 # Mean Strategy used for posterior mean
  - variance@posterior_variance: fixed_small    # Variance Strategy used for posterior variance
  - mean@model_mean: epsilon                    # Model Mean parametrization
  - variance@model_variance: fixed_small        # Model Variance parametrization
  - loss: mse_mean_epsilon_simple               # Loss used during the training
  - model: unet                                 # Architecture used during the training
```

The mean and variance strategies can be applied in multiple places within the diffusion process (e.g., posterior_mean vs. model_mean). To differentiate these, we use the `@` operator in overrides.

To modify model mean estimation, loss function, and variance strategy, follow the example:

```bash
uv run train.py \
   diffusion/mean@diffusion.model_mean=xstart \
   diffusion/variance@diffusion.model_variance=fixed_large \
   diffusion.loss vlb
```
#### 🚨 **No Configuration Restriction**
This framework does not enforce consistency between components. It allows any combination of settings, even if they are nonsensical or incompatible. Example: Using a loss function that expects `xstart` but parameterizing the mean as `epsilon`.

```bash
uv run train.py \
diffusion/mean@diffusion.model_mean=epsilon \
diffusion/loss=direct_mean
```
This setup will run, but the loss will likely be meaningless since `direct_mean` expects `xstart` parameterization.
